# 数据生命周期：读穿透、缺口补抓与复权处理

> 本文档描述数据从抓取到查询的运行时行为，是 [refactor-design.md](refactor-design.md)
> 5.4 节规则的操作视角说明。规则实现：`db/coverage.py`（30 个单元测试覆盖）。
> 接口清单见 [api.md](api.md)。

## 1. 每次查询的判定流程

API 收到数据请求后不直接查事实表，而是先查 `data_watermark` 水位表
（一次主键点查）做三段判定：

```
请求范围 vs 水位 [first_date, last_date] + last_fetched_at
│
├─ 已覆盖且新鲜 ──────────▶ 直接 SELECT 返回（毫秒级，不碰 baostock）
├─ 有缺口（范围超出水位）──▶ 只对缺口段投递抓取任务，等完成后读库
└─ 覆盖但未定型部分过期 ───▶ 只重抓过期的尾段（如今天的 bar）
```

核心原则：**数据一旦"定型"永久有效，永远不再重抓**。会触发抓取的只有：

| 情况 | 行为 |
|---|---|
| 查历史范围（昨天及以前）且已入库 | 永远读库，与抓取时间无关 |
| 范围含今天，距上次抓取 > 刷新间隔（K线/因子 5 分钟） | 只重抓未定型段 |
| 范围超出已覆盖区间 | 只补抓缺口（见第 2 节） |
| 财报披露窗口内（截止日未到），上次抓取 > 1 天 | 重抓该季度；截止日已过则永久 |
| 宏观数据在近 2 个月内，上次抓取 > 7 天 | 重抓沉淀期内段；更早的永久 |
| 周/月线：本周/本月的 bar | 按 5 分钟刷新；上周五/上月末收盘后定型 |

完整规则表（各数据集的定型边界与刷新间隔）见设计文档 5.4 节。

## 2. 缺口补抓：只抓差集

抓取粒度永远是"请求范围与水位的差集"，已入库部分既不重抓也不改写。

**示例**：库中已有某 code 2026-04-01 ~ 04-20 的日 K，请求 04-01 ~ 04-30：

```
请求 [04-01 ────────────── 04-30]
水位 [04-01 ──── 04-20]
                  └─ 尾部缺口 = [04-21, 04-30]   ← 唯一抓取的范围
```

1. 投递一个任务 `fetch_kline(code, 04-21, 04-30)`（参数即缺口）；
2. worker 只向 baostock 查这一段，upsert 落库，同事务推进水位
   `last_date = GREATEST(04-20, 04-30)`；
3. API 一次 SELECT 读出完整 04-01 ~ 04-30 返回（新老数据拼接，调用方无感）。

其他形态：头部缺口对称处理；头尾都缺为两段（相邻自动合并为一个任务）；
请求延伸到今天时，尾部缺口与"今天的刷新区"合并为一段。

**并发去重**：两个客户端同时触发同一缺口时，`fetch_task` 的部分唯一索引
（pending/running 状态按参数哈希唯一）保证只抓一次，后到者等待同一任务。

**首次触达全量回填**：水位不存在（该 code 首次被查）时，从回填起点
（日线=1990-12-19 上交所开市；分钟线=`STOCKDATA_MINUTE_BACKFILL_START`）
抓到请求尾——保证覆盖区间永远连续无空洞，单一水位行才足以描述覆盖。

**超大缺口切片**：缺口超过上限（日/周/月线与复权因子 10 年、分钟线 2 年，
见 `readthrough._SLICE_DAYS`）时切成多段升序连续任务：单任务时长有界，
每段落库即推进水位——中途失败后重试只补剩余段，全史回填具备断点续传。

**未来日期钳制**：除交易日历外，请求尾超过今天的部分直接忽略
（不存在的数据不产生缺口，避免反复空投任务）。

## 3. 除权除息：追加因子，零失效

**场景**：库中已有某 code 1 月 1 日的 K 线，4 月 7 日发生分红除权。

下次查询时 1 月 1 日的数据**不会重抓，记录零变化**——`kline` 表存的是
不复权真实成交价，分红改变的是换算口径而非历史成交价。除权后数据库的
全部变化是**追加**：

| 表 | 变化 |
|---|---|
| `kline` | 零变化（含全部历史行） |
| `adjust_factor` | 新增一行 `(code, 04-07, fore/back 因子)` |
| `dividend` | 查当年分红时新增明细行（当年 1 天新鲜期自动捕获） |

新因子行入库的两条路径（先到先得）：被动——下次复权查询发现因子水位
缺口时补抓；主动——当晚 17:10 `sync_tracked_codes` 自动刷新（正常运行下
事件当天收盘后即入库，无需等查询）。

对比旧 diskcache 架构：一次除权曾导致该 code 全部复权 K 线缓存失效重抓；
现架构的代价是因子表追加一行，其余零失效。

## 4. 复权价：读取时计算

**公式（已对 baostock 输出逐位验证）**：bar 复权价 = 不复权价 × 因子，
因子取除权日 ≤ bar 日期的**最近一次事件**（前复权用 fore、后复权用 back），
首个事件之前因子为 1。baostock 的前复权因子锚定固定基准——历史事件因子
不随新事件改变，因此旧日期的复权价是稳定的，新事件只影响其后的 bar。

**每次读取都现算，且这是刻意选择**：

- 成本可忽略：一只股票几十年除权事件仅 10~20 行，逐 bar 二分查找 + 5 次乘法，
  O(bar数 × log 事件数)。实测全年 242 根 bar 复权与不复权请求延迟无差异
  （计算 < 1ms，淹没在 I/O 中）；
- 预存复权价的代价：存储 ×3、除权日重写全部历史复权行（写放大 + 失效窗口）、
  失效机制回归——现算让写路径与失效逻辑彻底归零；
- `adjust_flag=3`（默认）完全不触发计算，连因子表都不查。

若未来出现高频批量复权读取（如回测引擎），优化方向是带因子版本号的结果
缓存（key 含 `max(divid_operate_date)`，除权自动失效），接口无需改动。

## 5. 边界 bar 的终值保障

盘中抓取的当日 bar 是抓取时刻的快照而非收盘终值。覆盖度规则把"昨天及
以前"视为定型，不会主动重抓它——兜底是每交易日 17:10 的
`sync_tracked_codes`：其增量范围**从水位 last_date 当天（含）开始**，
专门在当晚把盘中快照覆写为收盘终值。因此正常运行下不存在残留的半截 bar。

## 6. 定时同步任务

beat 调度（时区 Asia/Shanghai，"每交易日"由 `trade_calendar` 表判定，
节假日自动跳过）：

| 时间 | 任务 | 作用 |
|---|---|---|
| 每日 08:00 | `sync_calendar` | 当年 + 次年交易日历（捕获临时调整） |
| 每日 08:30 | `refresh_yesterday_list` | 补抓昨日股票列表（当日列表盘中不发布） |
| 交易日 17:00 | `sync_market` | 股票列表、三大指数成分股、行业分类 |
| 交易日 17:10 | `sync_tracked_codes` | 遍历水位表，对**所有已入库代码**增量补抓 K 线/复权因子至当日 |

`sync_tracked_codes` 的效果：任何被查询过一次的标的从此自动保持最新——
交易时段外的查询永远直接命中库。

**实现位置**：调度表在 `fetcher/app.py` 的 `beat_schedule`，任务实现在
`fetcher/beat.py`。`sync_tracked_codes` 的逻辑：查出 `data_watermark` 中
所有 `code != ''` 且数据类型为 K 线（日/周/月/分钟）或复权因子的水位行，
对每个 (code, data_type) 投递一个增量任务，范围 `last_date（含）→ 今天`；
任务经 crc32 分片路由，同 code 同类型落在同一 worker 进程串行执行。

**没有独立的"订阅列表"表，跟踪是隐式的**：哪些 code 被持续同步，完全由
`data_watermark` 中是否存在该 code 的水位行决定。code 首次被查询时读穿透
全量回填并写入水位，从此自动进入每日增量同步；无需也没有手工注册入口。
表结构见第 8 节附录。

## 7. 季度财报的特殊语义

季度财报不用区间水位（点状抓取配区间水位会虚假覆盖中间未抓取的季度），
而是两个点状事实的组合判定（`coverage.check_quarter`）：

- `financial_report` 中该报告期**是否已有行**；
- `fetch_task` 中该 (code, year, quarter) **最近一次成功抓取时刻**——
  兼作负结果记忆：披露截止日后查过且确实没有 → 永久空结果不再重查；
  披露期内查过没有 → 1 天内不重查（未披露是常态）。

## 8. 附录：水位表 data_watermark 结构

ORM 定义在 `db/models/meta.py`（`DataWatermark`），由 alembic 初始迁移建表，
等价 DDL：

```sql
CREATE TABLE data_watermark (
    code            VARCHAR(12)  NOT NULL DEFAULT '',  -- 全市场数据集用空串
    data_type       VARCHAR(24)  NOT NULL,             -- 数据类型枚举，如 k_d / k_w / adjust_factor
    first_date      DATE,                              -- 覆盖区间起点（可空）
    last_date       DATE         NOT NULL,             -- 业务水位：数据覆盖到哪天
    last_fetched_at TIMESTAMPTZ  NOT NULL,             -- 系统水位：最后一次抓取动作时刻
    PRIMARY KEY (code, data_type)
);
```

没有额外索引和约束，唯一索引即 `(code, data_type)` 复合主键。
双水位语义：`[first_date, last_date]` 是连续闭区间（"首次触达全量回填"
保证无空洞），判断覆盖范围；`last_fetched_at` 判断新鲜度（规则见
`db/coverage.py` 与设计文档 5.4 节）。

与 `fetch_task` 表的分工：`data_watermark` 记录**状态**（每个标的每类数据
覆盖到哪、何时抓的），`fetch_task` 记录**动作流水**（每次抓取的参数、
状态、起止时间，兼任务去重）。"系统跟踪过哪些 code"查前者即可。
